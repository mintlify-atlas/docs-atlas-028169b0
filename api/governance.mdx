The governance API provides the oversight and operational infrastructure of the Workforce Intelligence OS. Alerts deliver system signals to individual users; the audit log captures every sensitive state change as an immutable record visible only to administrators; the ops endpoints manage personal duties (daily task notes) and end-of-day reports with a structured review workflow. All paths are prefixed with `/api/v1` and require a valid JWT.

<Warning>
  The audit log is **Admin-only** and **immutable**. Records can never be deleted or modified through the API. Attempts by non-admin callers return `403 PERMISSION_ERROR`.
</Warning>

---

## Alerts

Alerts are system-generated notifications attached to a specific recipient user. Each alert has a status (`OPEN`, `RESOLVED`) and is ordered by creation time with a maximum of 100 returned per request.

### `GET /api/v1/alerts`

Returns up to 100 alerts addressed to the currently authenticated user, ordered newest-first.

**Auth:** Any authenticated user (`get_current_user`).

<ResponseField name="[]" type="array">
  Array of `AlertRead` objects.
  <Expandable title="AlertRead fields">
    <ResponseField name="id" type="string">UUID of the alert.</ResponseField>
    <ResponseField name="recipient_user_id" type="string">UUID of the user this alert belongs to.</ResponseField>
    <ResponseField name="status" type="string">Alert lifecycle status: `OPEN` or `RESOLVED`.</ResponseField>
    <ResponseField name="message" type="string">Human-readable alert message.</ResponseField>
    <ResponseField name="created_at" type="string">ISO-8601 UTC creation timestamp.</ResponseField>
    <ResponseField name="resolved_at" type="string | null">ISO-8601 UTC timestamp when the alert was resolved, or `null` if still open.</ResponseField>
  </Expandable>
</ResponseField>

```json Example Response
[
  {
    "id": "a1b2c3d4-e5f6-...",
    "recipient_user_id": "u9v8w7x6-...",
    "status": "OPEN",
    "message": "Attendance session not closed — auto check-out applied.",
    "created_at": "2025-01-20T18:05:00Z",
    "resolved_at": null
  }
]
```

---

### `PATCH /api/v1/alerts/{alert_id}/resolve`

Marks a specific alert as resolved. Only the alert's intended recipient may resolve it.

**Auth:** Any authenticated user. Must own the alert.

<ParamField path="alert_id" type="string" required>
  UUID of the alert to resolve.
</ParamField>

<ResponseField name="id" type="string" />
<ResponseField name="status" type="string">Will be `RESOLVED` after a successful call.</ResponseField>
<ResponseField name="resolved_at" type="string">UTC timestamp set at resolution time.</ResponseField>

<Warning>
  Attempting to resolve an alert owned by a different user returns `403 PERMISSION_ERROR`. A missing alert returns `404 NOT_FOUND`.
</Warning>

```json Example Response
{
  "id": "a1b2c3d4-e5f6-...",
  "recipient_user_id": "u9v8w7x6-...",
  "status": "RESOLVED",
  "message": "Attendance session not closed — auto check-out applied.",
  "created_at": "2025-01-20T18:05:00Z",
  "resolved_at": "2025-01-20T18:12:34Z"
}
```

## Audit Logs

The audit log is the tamper-evident record of sensitive operations performed in the system. Every destructive or privileged action — user role changes, permission overrides, data exports — is written here automatically by the application.

### `GET /api/v1/audit-logs`

Returns up to 200 audit log entries ordered by creation time (newest-first).

**Auth:** `ADMIN` role **required**. All other roles receive `403 PERMISSION_ERROR`.

<ResponseField name="[]" type="array">
  Array of `AuditLogRead` objects.
  <Expandable title="AuditLogRead fields">
    <ResponseField name="id" type="string">UUID of the log entry.</ResponseField>
    <ResponseField name="actor_id" type="string">UUID of the user who performed the action.</ResponseField>
    <ResponseField name="action" type="string">Action identifier string, e.g. `"user.role_changed"` or `"permission.override_set"`.</ResponseField>
    <ResponseField name="entity_type" type="string | null">The type of resource affected, e.g. `"user"`, `"department"`.</ResponseField>
    <ResponseField name="entity_id" type="string | null">UUID of the affected resource.</ResponseField>
    <ResponseField name="metadata" type="object | null">JSON object containing before/after state or additional context.</ResponseField>
    <ResponseField name="ip_address" type="string | null">IP address of the request origin.</ResponseField>
    <ResponseField name="created_at" type="string">ISO-8601 UTC timestamp of the action.</ResponseField>
  </Expandable>
</ResponseField>

<Note>
  Audit log entries are written by the server and cannot be created, edited, or deleted via the API. The 200-record cap applies per request.
</Note>

```bash Example Request
GET /api/v1/audit-logs
Authorization: Bearer <admin_token>
```

```json Example Response
[
  {
    "id": "log-1234-...",
    "actor_id": "admin-uuid-...",
    "action": "user.role_changed",
    "entity_type": "user",
    "entity_id": "emp-uuid-...",
    "metadata": {
      "previous_role": "employee",
      "new_role": "team_lead"
    },
    "ip_address": "192.168.1.10",
    "created_at": "2025-01-19T14:22:05Z"
  }
]
```

## System Health

### `GET /health`

A lightweight liveness probe that confirms the API process is running. This endpoint is mounted at the application root — **not** under the `/api/v1` prefix — so it is always reachable even if the versioned router is misconfigured.

**Auth:** None required. Suitable for load-balancer health checks.

<ResponseField name="status" type="string">Always `"ok"` when the process is running.</ResponseField>

```bash Example Request
GET /health
```

```json Example Response
{ "status": "ok" }
```

<Tip>
  Use this endpoint in Kubernetes `livenessProbe` and `readinessProbe` configurations. Permission seeding and admin bootstrapping run automatically at application startup — they do not require manual API calls.
</Tip>

## Daily Duties

Personal duties are lightweight daily task notes backed by `PersonalNote` records. They carry a `title`, optional `description`, and a `pending` / `completed` status. All duty endpoints operate on the currently authenticated user — there is no cross-user access.

### `GET /api/v1/duties`

Returns all duties logged by the authenticated user for today (Asia/Karachi date).

**Auth:** Any authenticated user.

<ResponseField name="[]" type="array">
  Array of `DutyRead` objects.
  <Expandable title="DutyRead fields">
    <ResponseField name="id" type="string">UUID of the duty (maps to the underlying `PersonalNote` ID).</ResponseField>
    <ResponseField name="user_id" type="string">UUID of the owning user.</ResponseField>
    <ResponseField name="title" type="string">Duty title extracted from the note content.</ResponseField>
    <ResponseField name="description" type="string | null">Optional additional detail.</ResponseField>
    <ResponseField name="status" type="string">`pending` or `completed`.</ResponseField>
    <ResponseField name="created_at" type="string">ISO-8601 UTC creation timestamp.</ResponseField>
    <ResponseField name="updated_at" type="string">ISO-8601 UTC last-updated timestamp.</ResponseField>
  </Expandable>
</ResponseField>

```json Example Response
[
  {
    "id": "d1e2f3a4-...",
    "user_id": "u1a2b3c4-...",
    "title": "Review PR #142",
    "description": "Check the authentication middleware changes",
    "status": "pending",
    "created_at": "2025-01-20T08:30:00Z",
    "updated_at": "2025-01-20T08:30:00Z"
  }
]
```

---

### `POST /api/v1/duties`

Creates a new duty for today.

**Auth:** Any authenticated user.

<ParamField body="title" type="string" required>
  Duty title.
</ParamField>

<ParamField body="description" type="string">
  Optional additional detail.
</ParamField>

<ResponseField name="id" type="string">UUID of the created duty.</ResponseField>
<ResponseField name="status" type="string">Always `"pending"` on creation.</ResponseField>

```json Example Request Body
{
  "title": "Write unit tests for attendance service",
  "description": "Cover check-in, check-out, and auto-checkout edge cases"
}
```

---

### `PATCH /api/v1/duties/{duty_id}`

Update a duty's title, description, or completion status.

**Auth:** Any authenticated user. Must own the duty.

<ParamField path="duty_id" type="string" required>
  UUID of the duty to update.
</ParamField>

<ParamField body="title" type="string">Updated title.</ParamField>
<ParamField body="description" type="string">Updated description.</ParamField>
<ParamField body="status" type="string">Set to `"completed"` to mark done, `"pending"` to reopen.</ParamField>

---

### `DELETE /api/v1/duties/{duty_id}`

Permanently deletes a duty.

**Auth:** Any authenticated user. Must own the duty.

<ParamField path="duty_id" type="string" required>
  UUID of the duty to delete.
</ParamField>

<ResponseField name="success" type="boolean">Always `true` on success.</ResponseField>

## End-of-Day Reports

EOD reports are structured daily summaries submitted by employees and reviewed by their direct manager. The workflow progresses through states: `Generated` → `Pending Approval` → `Approved` / `Rejected` / `Needs Revision`.

### `GET /api/v1/eod/me`

Returns the authenticated user's EOD report for a specific date, or `null` if none exists.

**Auth:** Any authenticated user.

<ParamField query="date" type="string">
  Report date in ISO format `YYYY-MM-DD`. Defaults to today (Asia/Karachi timezone).
</ParamField>

<ResponseField name="id" type="string | null">UUID of the EOD report, or response is `null`.</ResponseField>

---

### `GET /api/v1/eod/me/today`

Convenience alias for `GET /eod/me` with today's date. Returns the current user's EOD for today or `null`.

**Auth:** Any authenticated user.

---

### `POST /api/v1/eod/me/generate`

Generates (or refreshes) the authenticated user's EOD report for today by pulling attendance, time logs, and task activity. If a report already exists it is refreshed in place.

**Auth:** Any authenticated user.

<ResponseField name="id" type="string">UUID of the generated EOD report.</ResponseField>
<ResponseField name="status" type="string">Report status after generation, e.g. `"Generated"`.</ResponseField>

---

### `POST /api/v1/eod/me/submit`

Submits the authenticated user's EOD report for manager review.

**Auth:** Any authenticated user.

<ParamField body="report_date" type="string" required>
  Date of the report being submitted. ISO format `YYYY-MM-DD`.
</ParamField>

<ParamField body="work_summary" type="string">
  Free-text summary of work completed today.
</ParamField>

<ParamField body="blockers" type="string">
  Any blockers encountered.
</ParamField>

<ParamField body="next_day_plan" type="string">
  Plan for the following day.
</ParamField>

<ResponseField name="status" type="string">Will be `"Pending Approval"` after successful submission.</ResponseField>

<Warning>
  An EOD report must exist (via `generate`) before it can be submitted. Attempting to submit a non-existent report returns `422`.
</Warning>

---

### `GET /api/v1/eod/team`

Returns EOD reports for the manager's direct reports that have been submitted for review. Supports filtering by status, date, and employee search.

**Auth:** `MANAGER` or `ADMIN` role required.

<ParamField query="search" type="string">
  Filter by employee name, email, role, department, or designation.
</ParamField>

<ParamField query="status" type="string">
  Filter by EOD status: `Pending Approval`, `Approved`, `Rejected`, or `Needs Revision`.
</ParamField>

<ParamField query="report_date" type="string">
  Filter to a specific date ISO `YYYY-MM-DD`.
</ParamField>

<ResponseField name="[]" type="array">
  Array of `EODReportRead` objects for the manager's direct reports, ordered by date descending.
</ResponseField>

---

### `POST /api/v1/eod/{report_id}/review`

Reviews a submitted EOD report by approving, rejecting, or requesting revision.

**Auth:** `MANAGER` or `ADMIN` role required. The reviewer must be the submitter's direct manager.

<ParamField path="report_id" type="string" required>
  UUID of the EOD report to review.
</ParamField>

<ParamField body="action" type="string" required>
  Review decision. One of: `Approved`, `Rejected`, `Needs Revision`.
</ParamField>

<ParamField body="comments" type="string">
  Optional reviewer feedback attached to the report.
</ParamField>

<ResponseField name="id" type="string">UUID of the reviewed report.</ResponseField>
<ResponseField name="status" type="string">Updated status reflecting the review decision.</ResponseField>
<ResponseField name="manager_comments" type="string | null">Comments stored on the report.</ResponseField>

<Warning>
  A manager cannot review their own EOD report. Attempting to do so returns `403`. HR Operations users are not authorised to review EODs — only the direct manager or an Admin may review.
</Warning>

```json Example Request Body
{
  "action": "Approved",
  "comments": "Great work today — well-structured summary."
}
```

## Error Reference

| HTTP Status | Error Code | Reason |
|---|---|---|
| `401` | `AUTH_ERROR` | Missing or invalid JWT |
| `403` | `PERMISSION_ERROR` | Caller is not authorised (wrong role or not the owner) |
| `404` | `NOT_FOUND` | Alert, duty, or EOD report not found |
| `422` | `VALIDATION_ERROR` | Invalid review action or missing required fields |
| `500` | `INTERNAL_ERROR` | Unhandled server error |
