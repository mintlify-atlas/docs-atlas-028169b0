The Projects and Tasks APIs work in tandem to model the organisation's work breakdown structure. Projects act as containers that require management approval before tasks can be created under them. Once a project is approved, managers and admins can create tasks, assign them to employees, advance them through a defined status lifecycle, and set complexity scores for workload analysis. Interns submit completion requests that must be approved by a manager before a task is closed.

<Note>
  Project endpoints are prefixed `/api/v1/projects`; task endpoints are prefixed `/api/v1/tasks`. Both require an `Authorization: Bearer <token>` header.
</Note>

## Enum Reference

### Project Priority

| Value | Description |
|-------|-------------|
| `low` | Background or nice-to-have work. |
| `medium` | Standard delivery (default). |
| `high` | Important delivery with a near-term deadline. |
| `critical` | Blocking or time-sensitive — requires immediate attention. |

### Approval Status

| Value | Description |
|-------|-------------|
| `pending` | Submitted and awaiting a decision. |
| `approved` | Approved — tasks may now be created. |
| `rejected` | Declined; `rejected_reason` contains the explanation. |
| `escalated` | Escalated to a senior approver. |
| `cancelled` | Withdrawn before a decision was reached. |
| `needs_clarification` | Approver has requested more information. |

### Project Status

| Value | Description |
|-------|-------------|
| `draft` | Created but not yet submitted for approval. |
| `pending_approval` | Submitted and in the approval queue. |
| `approved` | Approved by management. |
| `active` | Actively in execution. |
| `on_hold` | Temporarily paused. |
| `completed` | All tasks done and project closed. |
| `rejected` | Not approved. |
| `archived` | Soft-deleted; excluded from default listings. |

### Task Status

| Value | Description |
|-------|-------------|
| `created` | Task created and awaiting assignment acknowledgement. |
| `approved` | Approved and ready to be started. |
| `in_progress` | Assignee is actively working on it. |
| `blocked` | Progress is blocked; `blocked_reason` is set. |
| `completed` | Assignee has marked it done. |
| `reviewed` | Manager has reviewed and signed off. |
| `reopened` | Sent back for rework after review. |
| `archived` | Soft-deleted; excluded from default listings. |

### Task Completion Request Status

| Value | Description |
|-------|-------------|
| `pending` | Request submitted, awaiting manager review. |
| `approved` | Approved — task is marked complete. |
| `rejected` | Rejected — task remains open. |
| `cancelled` | Cancelled before review. |
| `superseded` | Replaced by a newer request. |

## GET /projects

List projects visible to the authenticated caller. Admins and HR see all projects; Managers see projects they own or manage; Employees see projects they are involved with via tasks.

**Auth:** Any authenticated user. Results are RBAC-scoped.

### Query Parameters

<ParamField query="approval_status" type="string">
  Filter by approval status: `pending`, `approved`, `rejected`, `escalated`, `cancelled`, `needs_clarification`.
</ParamField>

<ParamField query="project_status" type="string">
  Filter by project lifecycle status: `draft`, `pending_approval`, `approved`, `active`, `on_hold`, `completed`, `rejected`, `archived`.
</ParamField>

<ParamField query="owner_id" type="uuid">
  Return only projects created by this user UUID.
</ParamField>

<ParamField query="manager_id" type="uuid">
  Return only projects assigned to this manager UUID.
</ParamField>

<ParamField query="include_archived" type="boolean">
  Include archived projects. Default: `false`.
</ParamField>

### Response Fields

<ResponseField name="[]" type="array">
  Array of ProjectRead objects.

  <Expandable title="ProjectRead">
    <ResponseField name="id" type="uuid" required>Unique project identifier.</ResponseField>
    <ResponseField name="title" type="string" required>Project name.</ResponseField>
    <ResponseField name="description" type="string">Optional project description.</ResponseField>
    <ResponseField name="owner_id" type="uuid" required>UUID of the user who submitted the project.</ResponseField>
    <ResponseField name="owner_name" type="string">Resolved display name of the project owner.</ResponseField>
    <ResponseField name="manager_id" type="uuid" required>UUID of the approving manager.</ResponseField>
    <ResponseField name="manager_name" type="string">Resolved display name of the manager.</ResponseField>
    <ResponseField name="priority" type="string" required>`low`, `medium`, `high`, or `critical`.</ResponseField>
    <ResponseField name="approval_status" type="string" required>See approval status table above.</ResponseField>
    <ResponseField name="project_status" type="string" required>See project status table above.</ResponseField>
    <ResponseField name="due_date" type="date">Target completion date (`YYYY-MM-DD`).</ResponseField>
    <ResponseField name="approved_at" type="datetime">Timestamp when the project was approved.</ResponseField>
    <ResponseField name="rejected_reason" type="string">Reason provided when the project was rejected.</ResponseField>
    <ResponseField name="progress_percentage" type="float">Computed percentage of completed tasks (0.0–100.0).</ResponseField>
    <ResponseField name="created_at" type="datetime" required>Creation timestamp.</ResponseField>
    <ResponseField name="updated_at" type="datetime" required>Last update timestamp.</ResponseField>
  </Expandable>
</ResponseField>

```bash Example
curl -X GET "https://api.example.com/api/v1/projects?approval_status=approved&project_status=active" \
  -H "Authorization: Bearer <token>"
```

## POST /projects

Submit a new project request for management approval. The project enters a `pending_approval` state until a manager or admin acts on it.

**Auth:** Any authenticated user.

### Request Body

<ParamField body="title" type="string" required>
  Project name. 1–255 characters.
</ParamField>

<ParamField body="description" type="string">
  Detailed project description.
</ParamField>

<ParamField body="priority" type="string">
  Urgency level: `low`, `medium` (default), `high`, or `critical`.
</ParamField>

<ParamField body="due_date" type="string">
  Target completion date in `YYYY-MM-DD` format.
</ParamField>

<ParamField body="manager_id" type="uuid">
  UUID of the manager who should approve this project. Optional for admins.
</ParamField>

### Response Fields

Returns the created `ProjectRead` with HTTP 201.

```bash Example
curl -X POST "https://api.example.com/api/v1/projects" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Customer Portal Redesign Q3",
    "description": "Full UI/UX overhaul of the self-service customer portal.",
    "priority": "high",
    "due_date": "2024-09-30",
    "manager_id": "a1b2c3d4-0000-0000-0000-000000000001"
  }'
```

## POST /projects/{project_id}/approve

Approve or reject a project that is pending approval. Only the assigned manager or an admin/HR operator may call this endpoint.

**Auth:** Manager (own projects), Admin, or HR Operations.

### Path Parameters

<ParamField path="project_id" type="uuid" required>
  UUID of the project to decide on.
</ParamField>

### Request Body

<ParamField body="decision" type="string" required>
  The outcome. Must be a valid `ApprovalStatus` value: `approved`, `rejected`, `pending`, `escalated`, `cancelled`, or `needs_clarification`.
</ParamField>

<ParamField body="reason" type="string">
  Optional explanation. Strongly recommended when `decision` is `rejected`.
</ParamField>

### Response Fields

Returns the updated `ProjectRead` with `approval_status` reflecting the decision.

<Tip>
  Once a project is approved, task creation against it becomes unlocked. Use `GET /projects/task-eligible` to fetch only projects available for task creation.
</Tip>

```bash Example
curl -X POST "https://api.example.com/api/v1/projects/3fa85f64-5717-4562-b3fc-2c963f66afa6/approve" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "decision": "approved"
  }'
```

## PATCH /tasks/{task_id}

**Auth:** Any authenticated user. Permitted transitions are RBAC-enforced.

### Path Parameters

<ParamField path="task_id" type="uuid" required>
  UUID of the task to update.
</ParamField>

### Request Body

<ParamField body="title" type="string">
  Updated task name. 1–255 characters.
</ParamField>

<ParamField body="description" type="string">
  Updated task description.
</ParamField>

<ParamField body="project_id" type="uuid">
  Reassign task to a different project.
</ParamField>

<ParamField body="assigned_to" type="uuid">
  Re-assign task to a different employee UUID.
</ParamField>

<ParamField body="parent_id" type="uuid">
  Move task under a different parent task.
</ParamField>

<ParamField body="priority" type="string">
  Updated priority: `low`, `medium`, `high`, or `critical`.
</ParamField>

<ParamField body="due_date" type="string">
  Updated due date in `YYYY-MM-DD` format.
</ParamField>

<ParamField body="status" type="string">
  New task status. Valid values and transitions depend on the caller's role.
</ParamField>

  Required when setting `status` to `blocked`. Explain the blocker.
</ParamField>

### Response Fields
